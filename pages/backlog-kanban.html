0. Cosa stai costruendo (in 20 secondi)

Un sistema dove:

  1. Scrivi l'idea in prompt.md

    La tua visione iniziale del progetto

  2. Copilot la converte in spec.md

    Specifica impeccabile, versionata come file reale

  3. Da spec.md generi task atomici

    Nel backlog, file reali e versionati

  4. Gestisci lo stato con Kanban

    CLI o Web UI integrata

  5. Copilot lavora task-by-task

    Su branch dedicata, con tracking completo

  6. Ricostruisci il contesto

    Da spec.md + backlog + board, senza memoria chat

Il Segreto

La chat è effimera, il repo è memoria.

1. Prerequisiti

  • Node.js + npm funzionanti
  • Repo Git inizializzato (git init) oppure esistente
  • Editor/IDE con Copilot (o Copilot CLI)

2. Installazione Backlog.md

Backlog.md è una CLI: puoi installarla globalmente (una volta sola sul PC) oppure localmente nel progetto (versionata nel repo).

Opzione A — Installazione globale (consigliata per uso personale)

Quando sceglierla

  • Vuoi usare Backlog.md su più progetti senza reinstallare
  • Vuoi digitare direttamente backlog ... nel terminale
  • Non ti interessa "bloccare" una versione specifica nel repository

Installazione:

npm install -g backlog.md

Verifica:

backlog --version

Uso:

cd /path/to/project
backlog init "NomeProgetto"

Opzione B — Installazione nel progetto (consigliata per team e CI/CD)

Quando sceglierla

  • Vuoi che il progetto "si porti dietro" la dipendenza
  • Vuoi usare Backlog.md in pipeline CI (GitHub Actions / GitLab CI)
  • Lavorate in team e volete evitare "a me funziona, a te no"

Prerequisito: Devi essere nella root del progetto (dove c'è package.json)

npm init -y # se package.json non esiste

Installazione:

npm install --save-dev backlog.md

Esecuzione:

npx backlog --version
npx backlog init "NomeProgetto"

3. Backlog Init (wizard) — cosa scegliere e perché

Nella root del progetto:

npx backlog init "NomeProgetto"

3.1 Scelta: AI Integration

Scegliere: via CLI commands (broader compatibility)

Cosa significa

Backlog.md non presume un editor MCP-ready, ti genera file di istruzioni "legacy" per far usare il backlog agli agenti tramite lettura file e comandi CLI.

Il flusso Spec-Driven funziona perfettamente anche con CLI integration.

3.2 "Select instruction files..."

Otterrai: .github/copilot-instructions.md

Perfetto: è il "contratto comportamentale" per Copilot.

3.3 Advanced Settings

Configurazione consigliata

  • Check task states across active branches: SI
  • Check task states in remote branches: SI
  • Active branch days: 30
  • Bypass git hooks when committing: NO
  • Auto commit: NO
  • Zero-padded IDs: 3 digits (task-001, task-002...)
  • Web UI port: 6420 (apri browser automaticamente)

Nota importante

"Bypass git hooks: yes" è comodo, ma in team può essere pericoloso (salti controlli). Per uso personale/locale va bene; in team valuta attentamente.

3.4 Shell completions failed (Windows)

Su Windows è normale vedere: Could not detect your shell...

Soluzione pragmatica:

  • Se usi Git Bash: --shell bash
  • Se usi WSL: --shell bash o zsh
  • Se non sai/ti cambia poco: ignora completions, non blocca nulla

4. Struttura "Spec-Driven" nel repo

Dopo init, crea questi file (se non esistono):

prompt.md
spec.md

E aggiungi due file guida nel backlog (consigliatissimi):

backlog/00_overview.md
backlog/ROADMAP.md

Backlog.md gestisce i task in backlog/tasks/..., ma aggiungiamo anche 00_overview.md e ROADMAP.md come "orchestrazione umana+AI".

5. Scrivere prompt.md (l'innesco)

Template pratico da copiare:

# Prompt

## Obiettivo
Cosa voglio ottenere (1-3 frasi).

## Contesto
- Repo: <nome>
- Vincoli: (compatibilità, performance, sicurezza)
- Parti interessate: (moduli, servizi, UI)

## Requisiti iniziali
- R1:
- R2:

## Non fare (Out of scope)
- OOS1:
- OOS2:

## Note
- Dipendenze
- Rischi

Questo file può essere grezzo.

6. Generare spec.md da prompt.md

Prompt corretto per Copilot (IDE Agent o CLI):

PROMPT da usare

Leggi il file prompt.md. Genera una specifica tecnica completa e salva l'output nel file spec.md nella root del progetto. Se spec.md esiste, sovrascrivilo completamente.

La spec deve includere:
1. Scopo
2. Ambito (In scope / Out of scope)
3. Requisiti funzionali numerati (RF1, RF2, ...)
4. Requisiti non funzionali (RNF1, ...: sicurezza, logging, performance)
5. Flussi / Use cases
6. Modifiche previste (componenti/file/aree)
7. Edge cases
8. Acceptance criteria testabili (AC1, AC2, ...)
9. Strategia di test (unit/integration/manual)

Non scrivere la spec in chat: scrivila SOLO nel file spec.md.

Risultato atteso: spec.md diventa la tua "verità tecnica".

7. Generare task ATOMICI da spec.md

Serve disciplina in due sottopassi:

7.1 Sottopasso A: Copilot produce una LISTA task

PROMPT

Leggi spec.md. Scomponi la specifica in task atomici (max 2-3 ore ciascuno).

Ogni task deve avere:
• Titolo breve
• Descrizione (2-6 righe)
• Acceptance criteria (2-6 punti, testabili)

Vincoli:
• Un task = una responsabilità
• Doc di riferimento: spec.md
• Evita task "mega"
• Indica eventuali dipendenze tra task

Crea un file dei tasks (così posso eseguirli uno per uno o creare uno script).
Il file chiamalo tasks-breakdown.md.

7.2 Sottopasso B: Copilot genera i COMANDI Backlog.md

PROMPT

Genera i comandi backlog task create per ogni task dal file tasks-breakdown.md.

Ogni comando deve includere:
• Titolo
• Descrizione (-d)
• Tutti gli acceptance criteria come --ac ripetuti
• Documento spec.md di riferimento (--doc spec.md)
• Altri parametri utili

Non eseguire i comandi: generali soltanto.

Metti tutti i comandi dei tasks in uno script unico eseguibile da PowerShell di Windows e dimmi come lanciarlo in chat.

A quel punto copi/incolli i comandi nel terminale, o esegui script generato.

8. Creare task in Backlog.md

Esempio di comando completo:

backlog task create "Implement authentication endpoint" ^
  -d "Create POST /auth/login according to spec.md (RF1, RF2). Include validation and error handling." ^
  --doc spec.md ^
  --ac "Returns 200 and a token on valid credentials" ^
  --ac "Returns 401 on invalid credentials" ^
  --ac "Logs authentication attempts without sensitive data"

Nota Windows

Su Windows CMD usa ^ per multilinea. Su PowerShell spesso funziona con backtick `. In alternativa scrivi tutto su una riga.

Backlog.md crea file tipo: backlog/tasks/task-001 - Implement authentication endpoint.md

9. Comandi Backlog.md essenziali

Board Kanban in CLI

backlog board

Avvia Web UI Kanban

backlog browser

Userà la porta che hai scelto (6420).

Lista task

backlog task list

Modifica task

backlog task edit 001 -s "In Progress"
backlog task edit 001 -s "Done"

(Opzionale) Riprova completions

backlog completion install --shell bash

10. Esecuzione con Copilot IDE (Agent Mode)

Dopo che spec.md e i task Backlog.md sono stati creati, la "palla" passa all'agente che deve gestire il ciclo di vita dei task in modo tracciabile.

Setup operativo

Apri (o rendi disponibili all'agente) questi elementi:

  • backlog task list --plain (stato globale)
  • backlog task <ID> --plain (dettaglio task scelto)
  • spec.md (contratto tecnico)
  • .github/copilot-instructions.md (regole)
  • backlog/00_overview.md (orchestrazione)

Algoritmo di selezione task (deterministico)

  1. Task in Progress

    Se esistono task in In Progress, riprende il più vecchio (o quello col numero più basso)

  2. Task To Do

    Altrimenti prende il primo task in To Do in ordine numerico

  3. Leggi plan/notes

    Prima di implementare, legge plan/notes già presenti

  4. In Progress + Plan

    Mette il task In Progress e inserisce un Implementation Plan

  5. Implementa

    Solo ciò che serve a soddisfare gli Acceptance Criteria

  6. Aggiorna Notes

    Aggiorna progressivamente le Implementation Notes

  7. Check AC

    Spunta gli AC man mano che vengono soddisfatti

  8. Final Summary

    Scrive un Final Summary con modifiche e test eseguiti

  9. Done

    Marca Done solo se tutti gli AC sono soddisfatti

Prompt "Master" (Backlog Task Manager)

PROMPT MASTER — Modalità Backlog Task Manager (Copilot Agent)

Sei un agente di sviluppo che lavora in modalità "Backlog Task Manager".

OBIETTIVO: Gestire e implementare i task Backlog.md in modo deterministico e tracciabile, 1 task alla volta, aggiornando sempre stato, plan, notes, acceptance criteria e final summary.

REGOLE NON NEGOZIABILI:
1) Lavora su UN SOLO task alla volta.
2) Non modificare file fuori dallo scope del task corrente.
3) Prima di scrivere codice:
   - identifica il task da prendere in carico seguendo l'algoritmo sotto
   - leggi il file del task e spec.md
   - scrivi un Implementation Plan nel task
4) Durante l'implementazione:
   - aggiorna Implementation Notes a step
   - spunta gli Acceptance Criteria appena soddisfatti
5) Alla fine:
   - esegui (o indica) i test/build necessari
   - scrivi Final Summary dettagliato
   - marca il task Done SOLO se tutti gli AC sono spuntati
6) Se trovi ambiguità:
   - NON inventare
   - aggiungi una nota nel task con la domanda

ALGORITMO DI SELEZIONE TASK:
A) Esegui backlog task list --plain
B) Se esiste un task In Progress: scegli quello con ID più basso
C) Altrimenti: scegli il primo task in To Do con ID più basso

PROCEDURA OPERATIVA:
1) Esegui backlog task N --plain
2) Metti il task In Progress: backlog task edit N -s "In Progress"
3) Se non esiste Implementation Plan, scrivilo
4) Implementa nel codice
5) Dopo ogni step: backlog task edit N --append-notes "..."
6) Quando soddisfi AC: backlog task edit N --check-ac X
7) A fine task: backlog task edit N --final-summary "..."
8) backlog task edit N -s Done

OUTPUT ATTESO:
- Sequenza chiara di comandi backlog + azioni nel codice
- Zero task lasciati "a metà" senza note
- Stato Kanban sempre aggiornato

Come usarlo: Apri Copilot Agent, incolla questo prompt, e poi digli: "Esegui un ciclo completo sul prossimo task".

11. Esecuzione con Copilot CLI

Copilot CLI è ottimo per:

  • Generare spec (scrivere spec.md)
  • Decomporre task
  • Fare review rispetto agli AC

Esempio "review":

Confronta le modifiche correnti con gli acceptance criteria del task-001. Elenca eventuali gap e suggerisci fix minimi.

12. Continuità tra chat/sessioni

Se domani apri una nuova chat e dici:

"Ripartiamo. Analizza spec e backlog e dimmi il prossimo task."

L'agente può:

  1. Leggere spec.md

    Contratto tecnico

  2. Leggere backlog/tasks/

    Stato e contenuti

  3. Leggere backlog/ROADMAP.md

    Ordine e progresso

Zero dipendenza dalla memoria chat

Tutto è ricostruibile dal repository.

13. Best Practices

I/O esplicito sempre

"scrivi in spec.md", "genera comandi backlog task create"

Task piccoli

2-3 ore massimo per task

AC testabili

Acceptance criteria chiari e misurabili

No scope creep

Copilot non rifattorizza "perché sì"

Spec prima del codice

Sempre documentare prima di implementare

Kanban aggiornato

Stato In Progress/Done sempre sincronizzato

Anti-pattern (cose da evitare)

  • "Implementa tutta la spec" — troppo generico
  • Task vaghi tipo "migliora performance" — non misurabile
  • AC non misurabili tipo "deve essere fatto bene" — ambiguo
  • Modifiche massive multi-modulo in un task — troppo grande
  • Non aggiornare lo stato Kanban — perdi tracciabilità

Mini-cheat-sheet dei comandi

# Init
npx backlog init "NomeProgetto"

# Task
backlog task create "Titolo" -d "Descrizione" --ac "AC1" --ac "AC2"
backlog task list
backlog task edit 001 -s "In Progress"
backlog task edit 001 -s "Done"

# Kanban
backlog board
backlog browser

# (opzionale)
backlog completion install --shell bash
